## Note de vérification

Une hypothèse de bug a été **invalidée** : `AggregateRoot::$domainEvents` est une propriété typée `array` non initialisée, mais PHP (≥ 7.4, testé en 8.4) auto-initialise le tableau lors du `$this->domainEvents[] = ...`. **Ce n'est donc PAS un bug.**

```php
// src/Shared/Aggregate/AggregateRoot.php
protected array $domainEvents;            // non initialisée -> OK quand même

public function recordDomainEvent(DomainEventInterface $event): self
{
    $this->domainEvents[] = $event;       // auto-init du tableau, pas d'erreur
    return $this;
}
```

## A. Problèmes d'architecture

### A1 🟠 Couplage BIDIRECTIONNEL entre Bounded Contexts (contredit le README)

Le README affirme : *« the BCs must not know anything about the other BCs »*. Or les deux contextes se référencent à la compilation via leurs classes d'événements.

```php
// src/Blog/User/Application/EventSubscriber/PublicationRequestedEventSubscriber.php:7
// (le User BC importe une classe du Post BC)
use App\Blog\Post\Article\Application\Event\OnPublicationRequestedEvent;
```

```php
// src/Blog/Post/Article/Application/EventSubscriber/OnUserVerifiedEventSubscriber.php:9
// (le Post BC importe une classe du User BC)
use App\Blog\User\Application\Event\OnUserVerifiedEvent;
```

**Cible :** contrats d'événements d'intégration dans un *shared kernel* neutre, ou messages sérialisés.

---

### A2 🟠 Modèle de domaine anémique

`Article` (et `User`) = sacs de getters/setters publics, aucune invariant protégée. Toute la « logique » se réduit aux factories `create()`.

```php
// src/Blog/Post/Article/Domain/Entity/Article.php:82-99
public function setBody(string $body): self
{
    $this->body = $body;
    return $this;
}

public function setTitle(string $title): self
{
    $this->title = $title;
    return $this;
}
// ... setCategory(), setAuthor(), setCreatedAt() ... tous publics
```

**Cible :** invariants dans l'entité, suppression des setters publics, comportements métier explicites.

---

### A3 🟠 La couche Application est polluée par le framework (adapters entrants mal placés)

Controllers, `ParamConverter`, Console Command, EventSubscribers (= adapters entrants) sont dans `Application/` au lieu de l'`Infrastructure`.

```php
// src/Blog/Post/Article/Application/Controller/Api/PostArticleController.php:9-12
use Symfony\Bundle\FrameworkBundle\Controller\AbstractController;
use Symfony\Component\HttpFoundation\JsonResponse;
use Symfony\Component\HttpFoundation\Request;
use Symfony\Component\Routing\Annotation\Route;
```

**Cible :** déplacer ces adapters en `Infrastructure/` ; `Application/` ne garde que des use-cases purs.

---

### A4 🟠 Le domaine connaît Symfony

**A4.1 — Les Domain Events étendent l'event Symfony :**

```php
// src/Blog/Post/Article/Domain/Event/ArticleCreatedEvent.php:9-11
use Symfony\Contracts\EventDispatcher\Event;

final class ArticleCreatedEvent extends Event implements DomainEventInterface
```

**A4.2 — L'entité domaine `User` implémente les interfaces de sécurité Symfony :**

```php
// src/Blog/User/Domain/Entity/User.php:10-13
use Symfony\Component\Security\Core\User\PasswordAuthenticatedUserInterface;
use Symfony\Component\Security\Core\User\UserInterface;

class User extends AggregateRoot implements UserInterface, PasswordAuthenticatedUserInterface
```

```yaml
# config/packages/security.yaml — provider branché directement sur l'entité domaine
providers:
    app_user_provider:
        entity:
            class: App\Blog\User\Domain\Entity\User
            property: email
```

**Cible :** `DomainEvent` neutre dans `Shared/` ; `User` domaine pur + adapter `SecurityUser` en Infra + `UserProvider` custom.

### A5 🟠 CQRS « logique » seulement

- **Un seul bus** par défaut, transports tous commentés → tout synchrone in-process.

```yaml
# config/packages/messenger.yaml (extrait — tout est commenté)
framework:
    messenger:
        transports:
            # async: '%env(MESSENGER_TRANSPORT_DSN)%'
        routing:
            # 'App\Message\YourMessage': async
```

- **Pas de read model** : la query lit les repos d'écriture et sérialise.
- **Commandes mutables** (setters) vs **query immuable** (constructeur) → incohérent.

```php
// src/Blog/Post/Article/Application/Model/CreateArticleCommand.php:22-25 (mutable)
public function setTitle(string $title): void { $this->title = $title; }
```

```php
// src/Blog/Post/Article/Application/Model/FindArticleQuery.php:11-14 (immuable)
public function __construct(string $articleId) { $this->articleId = $articleId; }
```

**Cible :** bus `command`/`query` séparés ; handlers renvoyant des DTO de lecture ; read model dédié.

### A6 🟠 Fuite d'abstraction dans un port (signature Doctrine dans le domaine)

```php
// src/Blog/Post/Article/Domain/Repository/CommentRepositoryInterface.php:9
public function findBy(array $criteria, array $orderBy = null, $limit = null, $offset = null);
```

**Cible :** méthode sémantique, ex. `findByArticleId(ArticleId $id): Comment[]`.

### A7 🟠 Sérialisation dans la couche application

Les handlers/services applicatifs injectent `SerializerInterface` et renvoient des `string` JSON.

```php
// src/Blog/Post/Article/Application/Service/ArticleFinderHandler.php:29-46
public function __invoke(FindArticleQuery $findArticleQuery): string
{
    // ...
    return json_encode($results, JSON_THROW_ON_ERROR);
}
```

**Cible :** retourner des DTO ; sérialiser dans le controller (couche présentation).

## B. Bugs / défauts concrets

### B1 🔴 Mots de passe stockés EN CLAIR

Aucun appel de hachage dans tout `src/` (vérifié par `grep`). Le mot de passe brut est stocké tel quel.

```php
// src/Blog/User/Domain/Entity/User.php:92-103
public static function registerUser(Email $email, array $roles, string $password): self
{
    $userId = Uuid::uuid4()->toString();
    $user = new User($userId);
    $user->setEmail($email->getValue());
    $user->setRoles($roles);
    $user->setPassword($password);   // <-- mot de passe brut, jamais haché
    // ...
}
```

```yaml
# config/packages/security.yaml — un hasher est configuré... mais jamais invoqué
password_hashers:
    Symfony\Component\Security\Core\User\PasswordAuthenticatedUserInterface: 'auto'
```

**Cible :** hacher via `UserPasswordHasherInterface` dans un adapter (jamais dans le domaine).

---

### B2 🔴 Canal de retour par la session HTTP (anti-pattern)

Le handler écrit son résultat dans la session ; le controller le relit ensuite.

```php
// src/Blog/Post/Article/Application/Service/CreateArticleHandler.php:48-53
$this->articleRepository->save($article);

$this->requestStack->getSession()->set(
    'last_article_created',
    $this->serializer->serialize($article, 'json')
);
```

```php
// src/Blog/Post/Article/Application/Controller/Api/PostArticleController.php:41-43
return JsonResponse::fromJsonString(
    $request->getSession()->get('last_article_created')
);
```

**Problèmes :** couple un handler applicatif à HTTP ; casse si le bus devient async/multi-process ; **race condition** entre deux créations concurrentes (la clé de session est écrasée).

**Cible :** le handler retourne un result/DTO (id créé) que le controller renvoie directement.

## C. Incohérences / dette

### C1 🟡 Deux styles d'entrée applicatifs incohérents

- **User BC** : Console Command → service applicatif appelé en direct (sans bus).
- **Post BC** : chaîne d'événements → message bus.

```php
// src/Blog/User/Application/Command/CreateUserCommand.php:51-55 (appel direct du service)
$user = $this->createUserService->handle(
    $input->getArgument('email'),
    [$input->getArgument('role')],
    $input->getArgument('password')
);
```

---

### C2 🟡 `CreateUserCommand` mal nommé/placé

C'est une **commande Console Symfony**, à ne pas confondre avec une commande CQRS (qui, côté Post, est dans `Application/Model/`).

```php
// src/Blog/User/Application/Command/CreateUserCommand.php:14-20
#[AsCommand(name: 'app:create-user', /* ... */)]
final class CreateUserCommand extends Command
```

---

### C3 🟡 `exclude` périmé dans `services.yaml`

Le chemin exclu n'existe plus (refactor « transform category into module ») ; le vrai chemin est `src/Blog/Post/Article/Domain/Entity/`. Les entités sont donc enregistrées comme services.

```yaml
# config/services.yaml
App\:
    resource: '../src/'
    exclude:
        - '../src/DependencyInjection/'
        - '../src/Blog/Post/Domain/Entity/'   # <-- chemin obsolète/inexistant
        - '../src/Kernel.php'
```

---

### C4 🟡 Aucun test

Pas de répertoire `tests/`, pas de `phpunit.xml`. Paradoxal pour une vitrine d'architecture testable.

**Cible :** tests unitaires domaine + intégration repository + fonctionnels API.

---

### C5 🟡 Value Objects incohérents / transients

`Article::getId()` renvoie un VO `ArticleId`, mais `Category::getId()` renvoie une `string`. De plus, les VO sont reconstruits à chaque appel (pas d'égalité d'objet).

```php
// src/Blog/Post/Article/Domain/Entity/Article.php:48-51
public function getId(): ?ArticleId
{
    return new ArticleId($this->id);     // nouveau VO à chaque appel
}
```

```php
// src/Blog/Post/Category/Domain/Entity/Category.php:26-29
public function getId(): string         // ici, string brute (incohérent)
{
    return $this->id;
}
```

## Points positifs (pour mémoire, à ne pas régresser)

- Découpage en couches Domain/Application/Infrastructure par module.
- Inversion de dépendance repository (port en Domain, adapter Doctrine en Infra).
- Mapping ORM externalisé en XML → entités domaine sans annotations.
- Références inter-agrégats **par identité** (`author`/`category` = `guid` scalaires) — conforme DDD.
- Value Objects validés à la construction (`AggregateRootId`, `EmailValueObject`).
- Encapsulation d'agrégat (`Article::createComment`).
- Chemin de lecture CQRS propre (`GetArticleController` → `FindArticleQuery`).
- `Category` immuable (constructeur, sans setters).
